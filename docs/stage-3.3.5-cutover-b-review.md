# Stage 3.3.5 Cutover B 设计评审

本阶段只做 Cutover 方案 B 的设计评审文档。不得进行功能开发，不得连接服务器，不得触发 Worker / RQ 任务，不得执行远程命令，不得修改真实数据库、真实日志、SSH Key、Passphrase、服务器密码或 `.env`。

## 当前链路现状

当前系统已经具备两条 active 中转线路：

- `hk-gost-test-8443`
  - `forwarding_method=gost`
  - 中转入口：`163.223.216.108:8443`
  - 目标：`74.211.97.116:443`
  - 状态：`active`
  - 当前定位：回退链路 / 保留
- `hk-socat-test-18443`
  - `forwarding_method=socat`
  - 中转入口：`163.223.216.108:18443`
  - 目标：`74.211.97.116:443`
  - 状态：`active`
  - 当前定位：测试可用链路 / 待正式 cutover

已完成的关键验收：

- `socat` 18443 客户端连通性已通过。
- Shadowrocket 复制原 Reality 节点，仅将 server 改为 `163.223.216.108`、port 改为 `18443` 后可正常上网。
- 本地 `nc -vz 163.223.216.108 18443` 已成功。
- `gost` 8443 仍保留，未被替换。
- `node.share_link` 未修改。
- Cutover 方案 A 已通过：前端可派生并复制 `socat` 18443 测试链接，但该链接仍被明确标记为测试链路。

## 方案 A / B / C 对比

| 方案 | 操作方式 | 是否改 `node.share_link` | 是否影响 `gost` 8443 | 是否新增端口 | 风险 |
| --- | --- | --- | --- | --- | --- |
| 方案 A | 前端派生并复制 `socat` 18443 测试链接 | 否 | 否 | 否 | 低 |
| 方案 B | 前端展示并复制 `socat` 18443 候选正式链接，但不写库 | 否 | 否 | 否 | 低到中 |
| 方案 C | 正式切换链路，例如写 `node.share_link` 或让 `socat` 接管 8443 | 可能是 | 可能是 | 否 | 高 |

方案 A 已完成，定位是“测试链接”。方案 B 仍不写正式数据，但会把已验证可用的 `socat` 18443 派生链接提升为“候选正式链接”展示，方便用户在客户端侧手动选择使用。方案 C 才涉及真正替换正式链路，应另行评审。

## 为什么先选方案 B

优先选择方案 B 的原因：

- `socat` 18443 已完成真实客户端连通性验收。
- 不需要新建远端服务，不需要新增监听端口。
- 不需要修改 `gost` 8443，回退链路保持可用。
- 不需要修改 `node.share_link`，避免误伤当前正式直连数据。
- 不需要数据库迁移，不需要新增字段。
- 不需要上传 SSH Key，不需要连接服务器，不触发任务。
- 失败回滚非常简单：用户停止使用候选链接即可，系统数据无需恢复。

## 方案 B 的边界

方案 B 只允许做前端候选正式链接展示和复制：

- 从当前 active node 的 `share_link` 派生候选链接。
- 仅替换地址为 `163.223.216.108`。
- 仅替换端口为 `18443`。
- UUID、flow、Reality security、SNI / serverName、publicKey、shortId、fingerprint、spiderX 等参数必须保持不变。
- 页面必须明确显示“候选正式链接 / 尚未写入系统正式链接”。
- 页面必须保留 `gost` 8443 回退链路提示。
- 页面必须提示：如果候选链接不可用，可继续使用原直连链接或 `gost` 8443 回退链路。

方案 B 严格禁止：

- 不写 `node.share_link`。
- 不修改 `transit_routes`。
- 不修改 `transit_routes.active` 或类似语义。
- 不删除、停用、重启、替换 `gost` 8443。
- 不把 `socat` 18443 自动设为正式线路。
- 不新增 Worker / RQ 任务。
- 不连接服务器。
- 不上传 SSH Key / Passphrase。
- 不执行远程命令。
- 不修改 Xray 配置、防火墙、iptables、nft、云安全组。
- 不生成二维码，不生成订阅链接，不做流量统计或测速。

## 是否修改 node.share_link

本阶段不修改 `node.share_link`。

方案 B 只生成前端本地候选链接，不写入数据库，不替换正式节点链接。`node.share_link` 仍保持原直连 Reality 链接，用于回退和对照。

## 是否新增数据库迁移

本阶段不新增数据库迁移。

现有 `transit_routes` 表已经可以表达当前两条 active route。方案 B 不需要新增字段，不需要新增表，不需要 Alembic。

## 是否新增监听端口

本阶段不新增监听端口。

方案 B 继续复用已验证的 `socat` 18443 测试链路，不创建新的 systemd service，不启动新进程，不开放新端口。

## 回滚方式

方案 B 的回滚方式：

- 不需要数据库回滚，因为不写库。
- 不需要远端恢复，因为不执行远端命令。
- 如候选链接不可用，用户停止使用候选链接即可。
- 页面继续保留 `gost` 8443 回退提示。
- `node.share_link` 仍保持原值，可继续复制原直连链接。
- `hk-gost-test-8443` 和 `hk-socat-test-18443` 两条 route 保持不变。

## Workbuddy 后续验证步骤

Workbuddy 后续验证应保持只读和前端验证优先：

1. 确认当前分支为 Stage 3.3.5 / Cutover B 相关分支。
2. 确认没有新增 Alembic 文件。
3. 确认没有新增后端 API。
4. 确认没有新增 Worker / RQ task type。
5. 确认没有改写 `node.share_link`。
6. 确认 active routes 仍为两条：
   - `hk-gost-test-8443 / gost / 8443 / active`
   - `hk-socat-test-18443 / socat / 18443 / active`
7. 打开“单条转发 -> 中转线路管理”。
8. 确认 `socat` 18443 显示候选正式链接区域。
9. 点击复制候选链接前应有确认提示。
10. 复制出的链接只改变 server 和 port，不改变 Reality 参数。
11. 不输出完整链接到报告、日志或文档。
12. 验证 pending / running tasks 为 0。
13. 验证 Redis `temp_credential:*` 为 0。
14. 验证 `/api/health` 全部 ok。
15. 验证没有服务器连接、远程命令、任务触发、数据库结构变化。

如需要真实客户端验证，应由用户在客户端侧手动导入候选链接；系统不得自动连接服务器或执行 cutover。
