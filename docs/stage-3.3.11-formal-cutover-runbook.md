# Stage 3.3.11 Formal Cutover Runbook

本阶段只写正式 cutover 执行手册。本阶段不是正式 cutover，不连接服务器，不执行远程命令，不触发任务，不修改业务代码，不修改正式节点链接。

## 当前阶段结论

- 本阶段只写 runbook。
- 当前尚未正式 cutover。
- `node.share_link` 未修改。
- `gost` 8443 仍是当前正式/回退链路。
- `socat` 18443 仍是候选正式链路。
- 本阶段不改变任何线路状态。

## 正式 cutover 可选路线

| 方案 | 目标 | 是否修改 `node.share_link` | 是否远程操作 | 是否影响 `gost` 8443 | 风险 |
| --- | --- | --- | --- | --- | --- |
| 方案 B | 继续只展示/复制 `socat` 18443 候选链接 | 否 | 否 | 否 | 低 |
| 方案 B+ | 记录客户端验收结果，增加 cutover readiness 状态，但不切换端口 | 待评审 | 否或待评审 | 否 | 中 |
| 方案 C | 正式切换，可能修改 `node.share_link`，可能让 `socat` 接管 8443 | 可能 | 可能 | 可能 | 高 |

### 方案 B

- 继续保持当前状态。
- 前端仍可展示和复制 `socat` 18443 候选正式链接。
- `node.share_link` 不写入候选链接。
- `gost` 8443 继续保留为当前正式/回退链路。
- 不需要服务器操作、数据库迁移或新增监听端口。

### 方案 B+

- 仍不让 `socat` 接管 8443。
- 仍不切换服务器端口。
- 可以考虑记录客户端验收结果、用户确认记录或 cutover readiness 状态。
- 如果需要持久化状态，必须单独评审数据库字段和 Alembic 迁移。
- 本 runbook 不实现 B+，只保留路线说明。

### 方案 C

- 进入真正正式 cutover。
- 可能修改 `node.share_link`。
- 可能让 `socat` 接管 8443。
- 可能停用、降级或保留 `gost` 8443。
- 需要远程服务器操作、回滚计划、客户端真实测试和明确操作窗口。
- 当前不建议直接进入方案 C。

## 正式 cutover 前置条件

正式 cutover 前至少需要满足：

1. Docker compose 或等价完整运行环境验收通过。
2. 客户端真实测试 `socat` 18443 通过。
3. 回退链路 `gost` 8443 确认可用。
4. 已明确是否修改 `node.share_link`。
5. 已明确是否让 `socat` 接管 `8443`。
6. 已明确是否停用、保留或降级 `gost` 8443。
7. 已准备回滚方案。
8. 已准备测试窗口和观察时间。
9. 已确认客户端可快速回退。
10. 已准备不包含完整节点链接或敏感凭据的验收记录。

## 8443 服务切换安全提醒

如果未来涉及 `8443` 服务切换，必须在执行前确认：

- 云服务器安全组 / 云防火墙已放行 TCP 8443。
- 服务器防火墙已放行 TCP 8443，例如 `ufw`、`firewalld`、`iptables` 或 `nftables`。
- 当前监听 `8443` 的进程已确认。
- 切换后 `8443` 将由哪个服务监听已确认。
- `gost` 8443 的回滚方式已保留。
- 客户端可在异常时快速切回旧链路。

## 未来正式 cutover 执行步骤草案

以下内容是 runbook 草案。当前阶段禁止执行。正式执行前必须二次确认，并且不得把真实服务器密钥、密码、完整节点链接或敏感 token 写入文档。

### 路线 1：继续方案 B

1. 保持 `node.share_link` 不变。
2. 保持 `gost` 8443 不变。
3. 继续在前端复制 `socat` 18443 候选链接。
4. 用户手动导入客户端新测试节点。
5. 如候选链接异常，继续使用 `gost` 8443 或原正式链路。

该路线无需远端命令，无需数据库迁移，无需新增监听端口。

### 路线 2：方案 B+ 状态记录

1. 单独评审是否需要数据库记录 cutover readiness。
2. 如果需要持久化，先设计字段、状态机、审计和 Alembic 迁移。
3. 只记录用户确认、候选链路验收和回退状态。
4. 不修改 `node.share_link`。
5. 不让 `socat` 接管 8443。

该路线必须单独开发和验收，不应混入正式 cutover。

### 路线 3：方案 C 正式切换

正式方案 C 必须单独开阶段，并在执行前完成二次确认。可能的执行步骤模板：

```text
当前阶段禁止执行以下模板。

1. 只读确认当前 active routes、active node 和回退链路。
2. 确认客户端已保存旧链接和候选链接。
3. 如果选择修改 node.share_link，先备份旧值。
4. 如果选择让 socat 接管 8443，先确认 8443 当前监听进程和防火墙放行状态。
5. 执行受控切换。
6. 验证监听、客户端连接、目标平台访问和短时间稳定性。
7. 如果任一验证失败，立即按回滚方案恢复。
```

不得在文档中写入真实 SSH Key、Passphrase、服务器密码、完整 `vless://` 链接、完整 token 或 Reality privateKey。

## 回滚方案

- 如果 `socat` 18443 客户端测试失败，继续使用 `gost` 8443 或原正式链路。
- 如果 `node.share_link` 修改后异常，应恢复旧值。
- 如果 `socat` 接管 `8443` 后异常，应恢复 `gost` 8443。
- 如果远端服务异常，应停止切换并恢复原状态。
- 如果云安全组或服务器防火墙导致 `8443` 不可达，应停止 cutover 并恢复回退链路。
- 回滚后必须再次验证客户端可用性。
- 回滚记录不得包含完整节点链接、SSH Key、Passphrase、服务器密码或 Reality privateKey。

## 验收清单

正式 cutover 验收至少需要确认：

- 前端显示正确。
- 候选链接复制正确。
- 客户端导入成功。
- 客户端连接成功。
- 普通网页访问成功。
- 目标平台访问成功。
- 短时间稳定性观察通过。
- 回退链路可用。
- 无误导性文案。
- `node.share_link` 是否修改已有明确记录。
- `gost` 8443 是否仍保留已有明确记录。
- `socat` 是否接管 8443 已有明确记录。
- 验收记录完整且不包含敏感信息。
