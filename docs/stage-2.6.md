# Stage 2.6 Notes

Stage 2.6 implements only frontend node export experience improvements.

Implemented:

- Mask the node detail `share_link` by default.
- Allow the admin to copy the full `vless://` share link.
- Allow the admin to show or hide the full `vless://` share link.
- Generate a QR code locally in the browser from the current node `share_link`.
- Keep the QR code hidden by default.
- Hide the full link and QR code when switching nodes.
- Add short client import hints for common clients.

Scope:

- Frontend only.
- No backend API changes.
- No database schema changes.
- No Alembic migration.
- No VPS connection.
- No SSH Key upload for this stage.
- No remote task is created or triggered by the export controls.

Share-link display policy:

- `GET /api/nodes` continues to hide `share_link` in the normal node list.
- `GET /api/nodes/{node_id}` remains the only node view that can provide
  `share_link`.
- The node detail page masks the link as the first 24 characters plus `...`
  plus the last 12 characters by default.
- The full link is shown only after the admin clicks the show button.
- Copying the full link does not require expanding the full link on screen.
- Deleted nodes remain hidden by default and their old share links are not
  displayed.

QR code policy:

- The QR code is generated by the browser from the full `share_link`.
- The QR code is equivalent to the full node link.
- The QR code is never uploaded to the backend.
- The QR code is not stored in the database.
- The QR code is not written to `task_logs`.
- The QR code is not written to backend logs.
- The QR code is hidden by default and is not persisted after page refresh.

Security notes:

- Reality privateKey must never enter frontend state, `result_data`,
  `task_logs`, or the database.
- `share_link` may be shown to the logged-in admin, but it is masked by default.
- Full `share_link` must not be added to ordinary task logs.
- The QR code should be treated as sensitive because anyone with the QR code can
  recover the full node link.

Not implemented:

- No backend QR-code image API.
- No subscription links.
- No batch export.
- No JSON export.
- No node creation.
- No node deletion.
- No node rebuild.
- No node refresh.
- No Xray restart.
- No VPS connection.
- No Xray config modification.
- No IEPL / IPLC relay.
- No `dokodemo-door`.
- No iptables forwarding.
- No firewall modification or port opening.
- No 3x-ui installation or API calls.
- No traffic statistics.
- No automatic speed tests.
- No SSH daemon configuration changes.

## Stage 2.6 Freeze Conclusion

Stage 2.6 is frozen after node export experience acceptance. This stage is a
frontend-only export experience improvement and did not change backend APIs,
database schema, Alembic migrations, VPS state, SSH credentials, Xray config, or
remote tasks.

Acceptance basis:

- Docker Compose services were normal.
- `/api/health` returned ok for backend, database, Redis, and Worker.
- Redis `temp_credential:*` count was 0.
- Pending/running task count was 0.
- `npm audit` reported `found 0 vulnerabilities`.
- The frontend returned HTTP 200 on `http://localhost:3000`.
- One active node, `direct-reality-recreated`, was present.
- `GET /api/nodes` and `GET /api/nodes/{node_id}` worked during acceptance.
- The node list showed the active node without exposing the full `share_link`.
- Node details showed node name, protocol, port, Reality fields, and flow, with
  no `reality_private_key` field.
- `share_link` was masked by default as the first 24 characters plus `...` plus
  the last 12 characters.
- The full link could be shown, hidden, and copied without first expanding it.
- Switching nodes hid the full link automatically.
- `react-qr-code@^2.0.21` was installed for frontend-only QR generation.
- The QR code was hidden by default, could be shown/hidden, was generated from
  the full `share_link`, and was small-screen safe.
- The QR code was not uploaded to the backend, not written to the database, and
  not written to task logs or backend logs.
- Short client import hints were available for generic clients, v2rayN,
  v2rayNG, Shadowrocket, and Passwall.
- Reality privateKey did not enter frontend state, `result_data`, `task_logs`,
  or the database.
- Task logs and backend/worker logs did not expose the full `share_link`.

Final allowed scope:

- Mask `share_link` in node details by default.

- Copy the full `share_link`.
- Generate QR codes locally in the browser.
- Show or hide QR codes.
