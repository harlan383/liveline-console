# Stage 3.3.8 Client Candidate Link Acceptance

本阶段只设计并记录 `socat` 18443 候选正式链接的客户端真实测试验收流程。本阶段不是正式 cutover，不修改正式节点链接，不连接服务器，不触发任务，不执行远程命令。

## 当前阶段结论

- 当前尚未正式 cutover。
- 本阶段只做客户端候选链接验收设计与记录。
- `gost` 8443 仍保留为正式/回退链路。
- `socat` 18443 仍是候选正式链路。
- `node.share_link` 未修改。
- 前端复制出的 `socat` 18443 候选正式链接只用于客户端手动导入测试。

## 客户端测试前置条件

开始客户端测试前必须确认：

- 前端页面可打开。
- “单条转发 -> 中转线路管理”可见。
- 页面明确显示当前尚未正式 cutover。
- 页面明确显示 `gost` 8443 仍保留为正式/回退链路。
- 页面明确显示 `socat` 18443 是候选正式链路。
- 可以复制 `socat` 18443 候选正式链接。
- 复制前确认提示已显示。
- 用户明确知道这不是正式切换。
- 原 `gost` 8443 或原直连链路仍可回退。
- 不在客户端中覆盖原正式节点，优先新建测试节点。

## 客户端测试步骤

1. 打开前端“单条转发 -> 中转线路管理”。
2. 找到 `hk-socat-test-18443` / `socat` / `18443` 候选正式链路。
3. 点击“复制 socat 18443 候选正式链接”。
4. 确认弹窗内容，确认这是候选链接且尚未正式 cutover。
5. 手动导入客户端。
6. 不覆盖原正式节点，建议新建一个测试节点。
7. 测试客户端连接。
8. 测试网页访问。
9. 测试目标平台访问。
10. 如用于直播线路，进行短时间稳定性观察。
11. 保留原 `gost` 8443 回退节点或原直连节点。
12. 如候选链路异常，立即切回回退节点。

测试时不得修改 `node.share_link`，不得停用 `gost` 8443，不得让 `socat` 接管 8443。

## 失败处理

如果 `socat` 18443 候选链路失败：

- 继续使用 `gost` 8443 或原正式链路。
- 不修改 `node.share_link`。
- 不停用 `gost` 8443。
- 不做正式 cutover。
- 不让 `socat` 接管 8443。
- 记录失败原因。
- 记录失败发生在哪一步：导入、连接、网页访问、目标平台访问、稳定性观察或回退。
- 如出现客户端本地网络异常，应区分本地代理规则、运营商网络、DNS、客户端配置和候选链路自身问题。

失败后应优先保持可用回退链路，而不是继续推进正式 cutover。

## 进入正式 cutover 前的准入条件

正式 cutover 前至少需要满足：

1. 候选链接客户端真实测试通过。
2. 回退链路确认可用。
3. 明确是否修改 `node.share_link`。
4. 明确是否让 `socat` 接管 `8443`。
5. 准备回滚计划。
6. 明确真实操作窗口，避免在直播或关键业务期间切换。
7. 如果涉及 `8443` 服务切换，必须确认云服务器安全组/云防火墙 TCP 8443 放行。
8. 如果涉及 `8443` 服务切换，必须检查服务器防火墙是否放行 TCP 8443。
9. 如果涉及数据库状态记录，必须单独评审 Alembic 迁移。
10. 如果涉及远端操作，必须单独评审允许命令和失败回滚。
11. 如果涉及写正式链接，必须明确旧链接保留和恢复方式。

未满足以上条件前，不得进入正式 cutover。
