## 方案 B / B+ / C 对比

| 方案 | 目标 | 是否改 `node.share_link` | 是否远程操作 | 是否新增迁移 | 是否新增端口 | 风险 |
| --- | --- | --- | --- | --- | --- | --- |
| 方案 B | 继续保持当前状态：展示并复制 `socat` 18443 候选链接 | 否 | 否 | 否 | 否 | 低 |
| 方案 B+ | 增加候选链路验收记录、用户确认记录或 cutover 准入状态 | 待评审 | 否或待评审 | 可能 | 否 | 中 |
| 方案 C | 正式 cutover，可能写正式链接或让 `socat` 接管 8443 | 可能 | 可能 | 待评审 | 否 | 高 |

### 方案 B

- 继续保持当前状态。
- `gost` 8443 保留为当前正式/回退链路。
- `socat` 18443 只作为候选正式链路展示和复制。
- 不修改 `node.share_link`。
- 不做远程操作。
- 回滚方式最简单：用户停止使用候选链接即可。

### 方案 B+

- 仍不切换服务器端口。
- 仍不让 `socat` 接管 8443。
- 可以考虑增加“候选链路验收记录”“用户确认记录”“cutover 准入状态”等管理能力。
- 如果需要持久化状态，必须单独评审数据库字段和 Alembic 迁移。
- 本阶段不实现 B+，只评审其方向。

### 方案 C

- 正式 cutover。
- 可能涉及修改 `node.share_link`。
- 可能涉及让 `socat` 接管 `8443`。
- 可能涉及停止、降级或删除 `gost` 8443。
- 需要远程服务器操作、真实客户端测试和回滚方案。
- 如果涉及 `8443` 服务切换，必须再次确认云服务器安全组/云防火墙 TCP 8443 放行，并检查服务器防火墙是否放行。
- 风险最高，不建议当前直接进入。

## 风险评估

### 用户侧风险

- 用户可能误以为候选链接已经正式切换。
- 用户可能复制候选链接后未进行客户端真实测试。
- 客户端本地代理规则可能污染 `nc` / `curl` 测试路径。
- 如果正式链接被过早替换，用户可能失去原可用链路。

### 远端服务器风险

- 方案 C 若操作 `8443`，可能影响现有 `gost` 回退链路。
- 远端 systemd 服务切换失败可能导致 8443 无监听。
- 云安全组或服务器防火墙未放行 TCP 8443 会造成客户端 timeout。
- 若误操作防火墙、iptables 或 nft，回滚复杂度会显著上升。

### 回滚风险

- 方案 B 回滚简单，不写库、不远程操作。
- 方案 B+ 若新增状态字段，需要设计状态回滚和审计。
- 方案 C 若修改 `node.share_link` 或替换 8443，需要准备旧链接、旧 route、旧 service 的完整恢复步骤。

### 数据库风险

- 方案 B 无数据库风险。
- 方案 B+ 可能需要新增字段或表，必须单独 Alembic 评审。
- 方案 C 若写 `node.share_link`，必须明确历史链接保留、审计和回滚策略。

### 文档/状态误导风险

- 不能把 `socat` 18443 写成已正式 cutover。
- 不能把 Docker compose 验收写成已通过。
- 不能让用户误以为 `gost` 8443 已停用。
- README、阶段文档和前端文案必须一致说明：当前尚未正式 cutover。

## 正式 cutover 的准入条件

正式 cutover 前至少需要满足：

1. Docker compose 或等价完整运行环境验收通过。
2. 前端候选链接复制后，客户端真实测试通过。
3. 确认 `socat` 18443 稳定性，包括连续使用、断线恢复、重启后状态。
4. 确认 `gost` 8443 回退仍可用，或明确它将被替换并准备回滚。
5. 明确是否修改 `node.share_link`。
6. 明确是否让 `socat` 接管 `8443`。
7. 准备可执行回滚计划。
8. 如果涉及 `8443` 服务切换，必须确认云服务器安全组/云防火墙 TCP 8443 放行。
9. 如果涉及 `8443` 服务切换，必须检查服务器防火墙是否放行 TCP 8443。
10. 明确是否需要 Alembic 迁移、审计记录或 cutover 状态字段。
11. 明确真实操作窗口，避免在业务直播或关键使用中操作。
12. 明确验收失败时不得删除 `gost` route，不得破坏原始节点链接。

## 推荐结论

基于当前状态，保守建议如下：

- 不建议直接进入方案 C。
- 建议继续保持方案 B 的低风险状态。
- 建议下一步先进入“客户端真实测试 / 候选链路验收记录”阶段。
- 如果需要记录验收状态或用户确认，应先评审方案 B+ 的数据模型，不要直接加迁移。
- 正式 cutover 应单独开新阶段，单独评审，单独授权。

当前推荐不修改 `node.share_link`，不关闭 `gost` 8443，不让 `socat` 接管 `8443`。
