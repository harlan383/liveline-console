# Stage 3.3.131 New Transit Worker Install Command Generation Approval

## Stage Goal

Stage 3.3.131 adds a no-real-VPS approval gate for future Worker install command
generation on `pending_worker` transit resources.

This stage builds on Stage 3.3.130. The operator can now review the exact
Go / No-Go items and typed confirmation that would be required before a later
stage is allowed to generate a one-time Worker token and real install command.

This stage does not generate a Worker token, does not generate a real install
command, does not install Worker, and does not execute remote actions.

## Implemented UI/API Changes

- The existing Worker install approval preview modal now includes a
  `生成命令前审批门` section.
- The gate shows the resource status, entry host readiness, SSH metadata
  readiness, public controller URL, and one-time token generation timing.
- The gate includes a typed confirmation input using:

```text
CONFIRM_GENERATE_WORKER_INSTALL_COMMAND_LATER
```

- Matching the typed confirmation only confirms the UI gate. It does not enable
  a token API call and does not create an executable command.
- The modal can copy a generation approval package that contains only
  placeholders and safety requirements.
- No backend API was added or changed.

## Command Generation Approval Gate Behavior

The approval gate records that a future command-generation stage must confirm:

- Resource status is `pending_worker`.
- Resource has an `entry_host`, or the value is explicitly marked as pending.
- SSH host / SSH port / SSH username are confirmed, or explicitly marked as
  pending for the next stage.
- Required Worker version is `0.1.24-stage-3.3.122`.
- Worker binary checksum is
  `cf7990f3ba0f85348fa714edb69a94d36b8752323fe9c843fa676cf50f38fcce`.
- Public controller URL is `http://my-con.golirong.xyz:8200`.
- Real install commands must use the public controller URL, not `localhost` or
  `127.0.0.1`.
- A real one-time Worker token can be generated only in a later approved stage.
- Any real token must not be stored in docs, README, PRs, logs, chat, or notes.

## Typed Confirmation

The typed confirmation text is:

```text
CONFIRM_GENERATE_WORKER_INSTALL_COMMAND_LATER
```

This is only a UI approval-gate rehearsal in this stage. Typing it does not
call a token endpoint, does not generate a command, and does not install Worker.

## Placeholder Token / Command Safety

The placeholder token remains:

```text
<generated-in-later-stage>
```

The placeholder command remains non-executable in this stage:

```text
curl -fsSL http://my-con.golirong.xyz:8200/worker/install.sh | \
  sudo bash -s -- \
  --controller-url http://my-con.golirong.xyz:8200 \
  --worker-token <generated-in-later-stage> \
  --role transit
```

No real token or real install command is generated by this stage.

## Go / No-Go Checklist

The approval package records:

- Generate one-time Worker token only in a later approved stage.
- Generate install command only in a later approved stage.
- Do not execute SSH or remote command in this approval stage.
- Do not store token in docs, README, PRs, logs, chat, or notes.
- Use public controller URL, not `localhost` or `127.0.0.1`.
- Confirm the real VPS can access the public controller.
- Confirm root/sudo, systemd, and curl are available.
